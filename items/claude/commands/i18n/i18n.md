description: "Integrate a feature/page into paraglide i18n. Finds hardcoded strings, adds keys to messages JSON files, and replaces strings with m.key() calls."
---

Integrate the feature or set of pages specified in `$ARGUMENTS` into paraglide i18n.

## Steps

### 1. Identify the target

**Mode A — explicit feature path** (when `$ARGUMENTS` is non-empty):

Parse `$ARGUMENTS` to determine the feature name and route path. Examples:
- `features/monitoring` → route `ui/src/routes/(app)/features/monitoring/`
- `infrastructure/cluster` → route `ui/src/routes/(app)/infrastructure/cluster/`
- `analytics/data-engine` → route `ui/src/routes/(app)/analytics/data-engine/`

If `$ARGUMENTS` is ambiguous, search for matching route directories and ask the user to confirm which one.

**Mode B — recently changed files** (when `$ARGUMENTS` is empty):

Look in the conversation context for a `--- file list ---` block. This block is injected automatically by the `UserPromptSubmit` hook when `/i18n` is typed without arguments, or it may be present from a prior `/recent` invocation.

Extract all file paths listed after `--- file list ---` and use those as the working set for steps 2–8. Skip route-path derivation — you already have the exact files. If no such block is found in context, ask the user to either provide a feature path or run `/recent` first.

### 2. Find affected files

Collect all `.svelte`, `.ts`, `.svelte.ts` files in:
- The route directory (recursively)
- Any component directories referenced by those files (e.g. `ui/src/components/<feature-name>/`)

Exclude files that are already fully i18n'd (i.e. contain no hardcoded UI strings).

### 3. Study naming conventions

Read `ui/messages/en.json` and identify the naming pattern used for the feature's prefix. Look at existing keys to determine:
- The prefix to use (e.g. `monitoring_`, `cluster_`, `registry_`)
- Existing keys that already cover some of the strings (reuse them, do not duplicate)
- Sub-namespaces like `monitoring_column_`, `monitoring_action_`, `monitoring_dialog_`, `monitoring_toast_`, `monitoring_form_`

Common sub-namespaces used across the codebase:
- `_title` — section/page/dialog titles
- `_description` — descriptive text
- `_placeholder` — input placeholders
- `_label_` — form/field labels
- `_column_` — table column headers
- `_action_` — button labels / actions
- `_toast_` — toast/notification messages
- `_dialog_` — dialog text
- `_empty` — empty state messages
- `_error` — error messages
- `_tab_` — tab labels

### 4. Extract hardcoded strings

For each affected file, identify all hardcoded UI-facing strings:
- Text content inside HTML/Svelte elements (e.g. `<h1>Create Alert</h1>`)
- String literals passed to props like `title="Create Alert"`, `placeholder="Search..."`
- String literals in JS/TS (e.g. `toast("Alert deleted")`, `label: "Name"`)
- Template literals with embedded variables should use paraglide's parameter syntax: `m.key({ param: value })`

Do NOT extract:
- CSS class names, IDs, or attribute values that are not user-visible
- Route paths or hrefs
- Enum values, API field names, or data identifiers
- Strings already using `m.some_key()`
- Developer-facing strings (console.log, error codes)

### 5. Add keys to messages files

For each extracted string, determine the appropriate key name following the conventions from step 3.

Update `ui/messages/en.json`:
- Add new keys in the appropriate alphabetical/logical position near related existing keys for that feature
- For strings with dynamic values, use paraglide parameter syntax: `"key": "Hello {name}"`

Update `ui/messages/de.json`:
- Add German translations for each new key
- Preserve grammatical correctness (capitalize nouns, use formal "Sie" where appropriate for UI)
- For strings with parameters, keep the same `{param}` placeholders

### 6. Replace hardcoded strings in source files

For each affected file:

1. Add the import if not already present:
   ```svelte
   import * as m from '$lib/paraglide/messages';
   ```
   Place it with other imports in the `<script>` block (or `<script lang="ts" module>` if the string is used there).

2. Replace each hardcoded string with the corresponding `m.key()` call:
   - In template HTML: `{m.monitoring_title()}`
   - In props: `title={m.monitoring_title()}`
   - In JS expressions: `m.monitoring_title()`
   - With parameters: `m.monitoring_greeting({ name: userName })`

3. For Svelte components that accept a `title` or `description` prop as a string, pass `{m.key()}` not the raw string.

### 7. Verify

After all replacements, quickly scan each modified file to confirm:
- No hardcoded UI strings remain
- All `m.key()` calls reference keys that exist in `en.json`
- Imports are correct

### 8. Clean up unused keys

Run:
```
cd ui && bun run fix:i18n
```

This removes any keys from the messages files that are no longer referenced in the codebase.

## Important rules

- **Never** add duplicate keys. Always check if a key already exists before adding.
- **Reuse** shared keys like `action_create`, `action_delete`, `action_cancel`, `action_save`, `search`, `status_*` etc. if they exist and match.
- **Do not** translate operator symbols, abbreviations, or technical identifiers.
- **Do not** modify `wt_*` (walkthrough) keys unless the walkthrough text itself needs updating — these are separate from UI strings.
- Match the exact formatting style of the existing JSON (2-space indent, no trailing commas).
- If a string appears in multiple files, add the key once and reuse it everywhere.
